Couchbase vs MySQL — Full Benchmark Report

Load test comparison across Insert, Get, Query & Consume endpoints · Backend (Grafana) metrics · March 2026

Introduction

This report compares Couchbase and MySQL as the backing datastore for the Product Variants Reader service. All scenarios were run on 10 application instances under four distinct workload types:

  • Insert Bulk — bulk insert of 2.5m records into each datastore.
  • Find by Primary Key — single-key lookups via the get endpoint.
  • Query with Limit 50 — read-only load against the query endpoint, simulating catalog read traffic.
  • Consume — a combined write workload per request: insert bulk of 50 records + patch bulk of 100 records + delete bulk of 50 records.

Environment

  • App pods: 4 CPUs, 6 GB memory each (JVM max heap: 3 GB)
  • Couchbase & MySQL: identical hardware, both configured with replication to another cluster
  • Dataset: 2.5 million records in each datastore
  • Metrics: Backend (Grafana) P95/P50 response times, CPU usage & RPM

Conclusions

Insert Bulk (2.5m records)

  • Both databases achieved similar throughput (~4.4K RPM).
  • MySQL had roughly half the P95 latency compared to Couchbase.

Find by Primary Key

  • Both reached the target (~597K avg RPM, ~800K peak).
  • At lower throughput both databases were comparable, but as RPM approached the 800K peak, Couchbase started exhibiting latency spikes — P95 reached up to 1.7s vs 208ms for MySQL.
  • Similar CPU usage (~24 cores).

Query with Limit 50

  • Both reached the target (~186K avg RPM, ~250K peak).
  • MySQL had lower latency (27ms vs 49ms P95 avg).
  • MySQL used less CPU (14.2 vs 18.3 cores).

Consume

  • Couchbase handled the load well: 163ms P95, 2.2K avg RPM.
  • MySQL collapsed: 33s P95 avg, only 453 avg RPM (vs 3K target), with spikes to 172s.
  • MySQL could not sustain the consume workload at this scale.

Overall

  • For pure read workloads (get, query), both databases are comparable at moderate throughput, but MySQL maintains more stable P95 latency under peak load.
  • For mixed read/write (consume), Couchbase dramatically outperforms — MySQL cannot sustain the target load.
  • Insert bulk throughput is comparable, with MySQL achieving lower latency.
  • Couchbase lacks native bulk operations, so the application must manage batching internally. This shifts CPU load to the application layer, which is especially noticeable in query-heavy workloads. For single read/write operations, Couchbase is expected to perform on par with MySQL.
